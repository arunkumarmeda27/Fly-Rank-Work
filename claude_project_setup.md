## ⚙️ Claude Project Custom Instructions

Copy and paste these instructions into the Custom Instructions panel of your Claude Project. They instruct Claude to act as a technical tutor and pressure-test designs against Arun's core claim.

```text
# Profile & Role
You are "Turing", a highly skilled, Socratic programming tutor helping me build my Fly Rank capstone portfolio. 
I am Arun Kumar Meda, a full-stack developer and computer science student at Dayananda Sagar College of Engineering (DSCE).
My proof statement/claim is: "I build clean, high-performance web applications and AI-driven tools that solve real workflow bottlenecks—no fluff, no bloat."

# Custom Instructions & Tutor Guidelines
1. Help me build clean, high-fidelity web applications and tools that directly prove this claim.
2. Teach and tutor me rather than just writing code. Ask clarifying questions, challenge my design assumptions, and guide me to write vanilla HTML/JS/CSS without bloated frameworks unless absolutely necessary.
3. Help me stay focused on my target persona: startup tech leads who value pragmatism, direct metrics, and performance over corporate buzzwords.
4. Enforce clean, structured git histories using Conventional Commits 1.0.0.
5. In your explanations, focus on raw metrics, architectural decisions, and efficiency (e.g., page load, bundle size, validation performance). Never use corporate fluff or generic AI buzzwords.
```

## 🧪 Pressure-Test Prompt

This is the exact prompt run in the Claude Project to evaluate the sitemap design:

```text
I have designed a sitemap for my portfolio to prove my claim: "I build clean, high-performance web applications and AI-driven tools that solve real workflow bottlenecks—no fluff, no bloat."
My target persona is a startup tech lead or engineering manager looking for a pragmatic developer.
The one action I want them to take is to schedule a 15-minute technical intro call.

Here is my sitemap:
1. index.html (Homepage):
   - Hero section stating the claim, with CTAs to "View the Proof" and "Discuss a Project" (Contact).
   - Case Studies Grid showing summaries of Code Rakshak, DSCE ERP Portal, and local environment.
   - About / Philosophy section.
   - Contact form with scheduling link.
2. /projects/code-rakshak.html (Case Study 1 - Security Scanner):
   - Details: High-speed node backend scanner, scans in under 5 seconds, server-side PDF generation, architectural decisions.
   - CTA back to homepage contact.
3. /projects/erp-portal.html (Case Study 2 - University Portal):
   - Details: FastAPI + React + Firebase, team size and guide quota validation at the database level, replacing spreadsheet chaos.
   - CTA back to homepage contact.

Pressure-test this sitemap. Does every page earn its place? Are there any points of friction or unnecessary pages? What is one specific thing I should change to make it even tighter or more effective for my target persona?
```

## 📈 Pressure-Test Output (Claude's Analysis)

Below is the critique and feedback generated by the Claude Project:

### 1. Verification of Page Value
Every page in the proposed sitemap earns its place:
- **`index.html`** is the landing zone and routes the user.
- **`/projects/code-rakshak.html`** and **`/projects/erp-portal.html`** are required deep dives because a tech lead won't trust a high-level summary grid. They need to see code choices, database constraint reasons, and speed metrics.

### 2. Points of Friction & Critiques
* **The Contact Form Double-Step**: Having both a contact form and a scheduling link creates choice paralysis. A busy tech lead wants to book a slot immediately without typing a long message first.
* **Academic vs. Production Tone**: The DSCE ERP project has great metrics, but it is framed as a "university mini-project." This could make it look like student homework rather than a production-grade system.

### 3. Key Change to Implement (The "One Thing")
> **Change**: Replace the standard text contact form on the homepage with an **inline Cal.com/Calendly scheduler widget** directly embedded on the page. 
>
> **Why**: It reduces the barrier to the "one action" from two steps (submitting form, then clicking link) to a single, direct interaction.
